## 1. บทนำ (Introduction)
### 1.1 วัตถุประสงค์ (Purpose)
เอกสารฉบับนี้จัดทำขึ้นเพื่อกำหนดความต้องการของระบบจัดการห้องสมุด ซึ่งเป็นระบบเว็บสำหรับบริหารจัดการการจองหนังสือ การตรวจสอบสถานะหนังสือ และการจัดการค่าปรับกรณีคืนหนังสือล่าช้า

### 1.2 ขอบเขตของระบบ (Scope)
ระบบนี้เป็นระบบ Web Application ที่ให้บริการลูกค้าในการค้นหา จอง และตรวจสอบสถานะหนังสือ รวมถึงให้ผู้ดูแลระบบจัดการข้อมูลหนังสือ ผู้ใช้ และค่าปรับ

### 1.3 คำจำกัดความ (Definitions)
* **Customer (ลูกค้า / นักศึกษา):** ผู้ใช้ทั่วไปที่สามารถค้นหาและจองหนังสือ
* **Professor (อาจารย์):** ผู้ใช้งานที่มีสิทธิ์การยืมหนังสือนานกว่านักศึกษา
* **Librarian (แอดมิน / เจ้าหน้าที่):** ผู้ดูแลระบบที่สามารถจัดการข้อมูลทั้งหมดในระบบ
* **JWT (JSON Web Token):** มาตรฐานการสร้าง Token สำหรับ Authentication

### 1.4 Problem Statement (ระบุปัญหาหลัก)
จากสถานการณ์ปัจจุบันของห้องสมุด สามารถระบุปัญหาหลักที่ระบบใหม่ต้องแก้ไขได้ ดังนี้:
1. **ปัญหาด้านการเข้าถึงข้อมูล (Accessibility):** ผู้ใช้บริการไม่สามารถตรวจสอบสถานะของหนังสือ (Available/Borrowed) ได้ด้วยตนเองจากระยะไกล ต้องเดินทางมาที่ห้องสมุดเท่านั้น ซึ่งก่อให้เกิดการเสียเวลาและความไม่พึงพอใจ
2. **ปัญหาด้านประสิทธิภาพการดำเนินงาน (Operational Efficiency):** ระบบการทำงานเดิมที่พึ่งพาไฟล์ Excel และสมุดบันทึก ไม่รองรับปริมาณผู้ใช้งานจำนวนมาก (2,500 คน) และหนังสือจำนวนมาก (15,000 เล่ม) ทำให้เกิดคอขวดในการให้บริการและการทำรายงานสถิติ
3. **ปัญหาด้านการสื่อสารและการแจ้งเตือน (Communication):** ขาดระบบแจ้งเตือนอัตโนมัติสำหรับการคืนหนังสือและการจอง ทำให้เกิดปัญหาหนังสือค้างส่งและผู้ที่ต้องการใช้หนังสือไม่ได้รับหนังสือตามเวลาที่ต้องการ
4. **ปัญหาด้านการจัดการสินทรัพย์ (Inventory Management):** การจัดการหนังสือใหม่และการตรวจสอบหนังสือสูญหายทำได้ยากเนื่องจากไม่มีระบบติดตามที่แม่นยำ

### 1.5 Pain Points (ปัญหาและความยุ่งยากที่พบ)
ปัญหาแบ่งตามกลุ่มผู้ใช้งานหลักได้ดังนี้:
* **สำหรับนักศึกษา (Student):**
  * **เสียเวลาเดินทางฟรี:** เดินทางมาห้องสมุดแล้วพบว่าหนังสือที่ต้องการถูกยืมไปแล้ว ทำให้เสียเวลาและต้องเดินทางกลับมาอีกครั้ง
  * **การรอคอยที่ยาวนาน:** ต้องรอคิวค้นหาหนังสือและทำรายการยืมนาน โดยเฉพาะช่วงสอบที่มีคนใช้บริการเยอะ
  * **ภาระค่าปรับ:** ลืมวันครบกำหนดคืนหนังสือเพราะไม่มีระบบแจ้งเตือน ทำให้ต้องเสียค่าปรับและถูกระงับสิทธิ์การยืม
  * **ความไม่แน่นอน:** ไม่รู้ว่าหนังสือที่ถูกยืมไปจะคืนเมื่อไหร่ ทำให้วางแผนการอ่านหนังสือไม่ได้
* **สำหรับเจ้าหน้าที่ห้องสมุด (Librarian):**
  * **กระบวนการทำงานล่าช้า:** การค้นหาข้อมูลในไฟล์ Excel ที่มีหลายพันรายการใช้เวลานาน (10-15 นาทีต่อคนในบางครั้ง)
  * **ความผิดพลาดและภาระงาน:** ต้องคำนวณค่าปรับด้วยตนเองและบันทึกข้อมูลซ้ำซ้อน ทำให้เกิดความผิดพลาดได้ง่ายและเกิดความเครียดในช่วงคนเยอะ
  * **การจัดการทรัพยากรยากลำบาก:** การจัดทำรายงานสถิติต้องใช้เวลารวบรวมข้อมูลทั้งวันจากหลายแหล่ง และการค้นหาหนังสือบนชั้นทำได้ยากเมื่อหนังสือวางผิดที่
* **สำหรับอาจารย์ (Professor):**
  * **วางแผนการสอนไม่ได้:** ต้องการใช้หนังสือประกอบการสอนแต่นักศึกษายืมไปแล้ว ทำให้แผนการสอนต้องสะดุด
  * **ขาดระบบจอง:** ไม่มีระบบจองหนังสือล่วงหน้า ต้องคอยมาเช็คที่ห้องสมุดบ่อยๆ หรือฝากเจ้าหน้าที่ซึ่งไม่มีประสิทธิภาพ
  * **ระยะเวลาการยืมไม่เพียงพอ:** สิทธิ์การยืมเท่ากับนักศึกษา ซึ่งไม่เพียงพอต่อการเตรียมสอนที่ต้องใช้เวลานานกว่า

## 2. ภาพรวมของระบบ (Overall Description)
### 2.1 กลุ่มผู้ใช้งานเป้าหมาย (Personas / User Classes)
1. **Persona 1: น้องมิ้นท์ (นักศึกษา)**
   * **บทบาท:** นักศึกษาปี 2 สาขาวิทยาการคอมพิวเตอร์
   * **พฤติกรรม:** ใช้มือถือเป็นหลัก ชอบความสะดวกรวดเร็ว คุ้นเคยกับแอปพลิเคชันอย่าง Shopee/LINE
   * **ความต้องการหลัก:** เช็คได้ว่ามีหนังสือหรือไม่ก่อนเดินทางมาห้องสมุด, ยืมหนังสือได้รวดเร็ว ไม่ต้องรอคิวนาน, มีระบบแจ้งเตือนก่อนครบกำหนดคืน
   * **Pain Point หลัก:** มาเก้อเพราะหนังสือหมด, ลืมคืนจนโดนค่าปรับ
2. **Persona 2: พี่ปาล์ม (เจ้าหน้าที่ห้องสมุด)**
   * **บทบาท:** เจ้าหน้าที่ห้องสมุดประจำ ประสบการณ์ 8 ปี
   * **พฤติกรรม:** ใช้ Excel ทำงาน แต่ไม่เชี่ยวชาญเทคโนโลยีขั้นสูง ชอบระบบที่ใช้งานง่าย
   * **ความต้องการหลัก:** ลดเวลาในการค้นหาข้อมูลและให้บริการ, ระบบคำนวณค่าปรับอัตโนมัติ, Export รายงานสถิติได้ง่าย
   * **Pain Point หลัก:** ทำงานไม่ทันช่วงสอบ, เครียดจากการคำนวณเงินและคิวยาว, เสียเวลาทำรายงานสิ้นเดือนนานมาก
3. **Persona 3: อาจารย์ ดร.สมชาย (อาจารย์ประจำ)**
   * **บทบาท:** อาจารย์ประจำภาควิชาวิทยาการคอมพิวเตอร์
   * **พฤติกรรม:** มีความรู้เทคโนโลยีสูง ชอบวางแผนล่วงหน้า
   * **ความต้องการหลัก:** ระบบจองหนังสือล่วงหน้าเพื่อเตรียมสอน, ระยะเวลาการยืมที่นานกว่านักศึกษา, ความมั่นใจว่าจะได้หนังสือตามแผน
   * **Pain Point หลัก:** วางแผนสอนแล้วหนังสือถูกยืมตัดหน้า, ไม่มีระบบจองทำให้ต้องเปลี่ยนแผนการสอน

### 2.2 สภาพแวดล้อมและข้อจำกัด (Environment & Constraints)
* **Environment:** ระบบทำงานผ่าน Web Browser รองรับการใช้งานบนคอมพิวเตอร์และอุปกรณ์พกพา
* **Dependencies:** ต้องเชื่อมต่ออินเทอร์เน็ต และต้องมีฐานข้อมูลสำหรับจัดเก็บข้อมูลหนังสือและผู้ใช้งาน

## 3. ความต้องการด้านฟังก์ชัน (Functional Requirements)

### FR-001: ค้นหาหนังสือ (Search Books)
* **Priority:** Must Have
* **Traceability:** `UR-01` (ต้องการรู้สถานะหนังสือก่อนเดินทาง) -> `FR-001`
* **Input:** คำค้นหา (Title, Author, Category)
* **Processing:** Query ฐานข้อมูลตาราง `books` ด้วยเงื่อนไขที่ตรงกับ Input
* **Output:** รายการหนังสือพร้อมสถานะ (ว่าง/ถูกยืม/ถูกจอง)
* **Acceptance Criteria:**
  * `Given`: ผู้ใช้อยู่ที่หน้าค้นหา
  * `When`: พิมพ์ชื่อหนังสือและกดค้นหา
  * `Then`: ระบบแสดงรายการหนังสือพร้อมระบุสถานะปัจจุบัน

### FR-002: การจองหนังสือ (Reserve Book)
* **Priority:** Must Have
* **Traceability:** `UR-02` (ต้องการจองหนังสือล่วงหน้า) -> `FR-002`
* **Input:** `book_id`, `user_id` (จาก Session)
* **Processing:** ตรวจสอบสถานะหนังสือ บันทึกการจอง และเปลี่ยนสถานะหนังสือ
* **Output:** ข้อความแจ้งเตือนสำเร็จ/ล้มเหลว
* **Acceptance Criteria:**
  * `Given`: ลูกค้าเข้าสู่ระบบ และเจอหนังสือสถานะ 'ว่าง'
  * `When`: กดจองหนังสือ
  * `Then`: ระบบบันทึกการจอง และเปลี่ยนสถานะเป็น 'ถูกจอง'

### FR-003: ระบบคำนวณค่าปรับและแจ้งเตือน (Fine & Notification)
* **Traceability:** `BR-01` (ลดภาระงานเจ้าหน้าที่) -> `FR-003`
* **Input:** วันที่ปัจจุบัน (System Date), รายการ borrowings ที่ status = 'active'
* **Processing:** หากลูกค้าไม่คืนหนังสือตามกำหนด ระบบจะคำนวณค่าปรับตามจำนวนวันที่เกินกำหนด โดยทำงานรายวัน (Cron Job)
* **Output:** ยอดค้างชำระในบัญชีผู้ใช้ และอีเมลแจ้งเตือน
* **Acceptance Criteria:**
  * `Given`: มีรายการยืมที่เกินกำหนด
  * `When`: ระบบรันการตรวจสอบรอบวัน
  * `Then`: คำนวณค่าปรับตามอัตราที่กำหนด บันทึกยอดค้างชำระ และส่งอีเมลแจ้งเตือน

## 4. ความต้องการที่ไม่ใช่ด้านฟังก์ชัน (Non-Functional Requirements)
* **Scalability:** ระบบต้องรองรับผู้ใช้งานพร้อมกันได้อย่างน้อย 100 คน และรองรับข้อมูลหนังสือได้ 15,000 เล่ม
* **Performance:** หน้าเว็บต้องโหลดภายใน 3 วินาที และการค้นหาหนังสือ 15,000 เล่มต้องแสดงผลภายใน 2 วินาที
* **Security:** ระบบต้องมีระบบรักษาความปลอดภัย เช่น การเข้ารหัสรหัสผ่าน ด้วย bcrypt และใช้ JWT สำหรับ Authentication
* **Availability:** ระบบต้องสามารถสำรองข้อมูลได้ อย่างน้อยวันละ 1 ครั้ง และเป้าหมาย Uptime ที่ 99.9% (downtime ไม่เกิน 8.76 ชม./ปี)
* **Usability:** ระบบต้องรองรับการใช้งานแบบ Mobile-First สำหรับผู้ใช้ (Student/Professor) และสามารถใช้งานได้อย่างเชี่ยวชาญ (Learnability) ภายใน 5 นาทีโดยไม่ต้องอบรม

## 5. Data Model / Database Schema

*ใช้ PostgreSQL ในการจัดการฐานข้อมูล*
```sql
CREATE TABLE users (
    user_id SERIAL PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    role VARCHAR(20) CHECK (role IN ('student', 'professor', 'librarian')) NOT NULL,
    email VARCHAR(100) NOT NULL UNIQUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE books (
    book_id SERIAL PRIMARY KEY,
    title VARCHAR(255) NOT NULL,
    author VARCHAR(150),
    category VARCHAR(100),
    status VARCHAR(20) CHECK (status IN ('available', 'borrowed', 'reserved')) DEFAULT 'available',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE borrowings (
    borrowing_id SERIAL PRIMARY KEY,
    user_id INT REFERENCES users(user_id) ON DELETE CASCADE,
    book_id INT REFERENCES books(book_id) ON DELETE CASCADE,
    borrow_date DATE NOT NULL,
    due_date DATE NOT NULL,
    return_date DATE,
    fine_amount DECIMAL(10, 2) DEFAULT 0.00,
    status VARCHAR(20) CHECK (status IN ('active', 'returned', 'overdue')) DEFAULT 'active'
);
